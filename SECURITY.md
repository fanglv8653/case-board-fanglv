# 安全政策

## 报告安全漏洞

**请不要通过 GitHub Issues 公开报告安全漏洞。**

如果你发现 CaseBoard 中的安全漏洞(尤其是涉及用户案件数据、本地文件访问、MCP 接入授权等敏感面的问题),请通过以下渠道私下报告:

- 🔒 通过 GitHub 私密漏洞报告(推荐):本仓库 **Security → Report a vulnerability**

请在报告中尽可能提供:

- 漏洞描述与潜在影响
- 复现步骤(操作系统版本、CaseBoard 版本、操作序列)
- 你认为可能的修复方向(可选)
- 是否希望在公告中被致谢

我们会在 **5 个工作日内**响应初步确认,并在合理时间内给出修复时间表。

## 支持的版本

V0.1 之前的开发版本均不提供安全维护。
正式发布的版本会维护最近一个 minor 版本的安全更新。

| 版本 | 安全更新 |
|:---|:---|
| < 0.6 | ❌ |
| 0.6.x(当前) | ✅ |

## 安装包与更新签名

- Tauri updater 的 `.sig` 是 minisign 更新制品签名，应用使用内置公钥验证最终安装包字节。
- updater minisign **不是** Windows Authenticode，不提供 Windows “已验证发布者”身份或 SmartScreen 信誉。
- 当前发布配置未包含 Authenticode 证书、指纹与时间戳，因此不得把安装包描述为“Windows 代码签名已验证”。请只从项目 Release 获取安装包，并核对发布页的哈希与 updater 签名信息。

## 数据隐私边界

CaseBoard 的设计原则:

- **本地存储**: 抽取后的结构化数据、报告、数据库均存在用户本机 SQLite / 文件
- **原文件只读**: 永不修改用户原始案件文件夹
- **云端处理须知**: 默认的文本抽取 / OCR 会把文档内容发往 DeepSeek / MinerU 处理(可改用本机模型);这是功能所需,非隐私缺陷
- **应用会写入本地数据**: 应用会按功能需要写入本地 SQLite、设置、缓存、抽取文本、报告、生成文书及用户主动执行的导入导出结果
- **反馈通道**: 反馈只在用户点击生成后写入本地 MD；自动诊断使用聚合计数、去标识文件类型和脱敏日志。应用不会在用户未确认时发送反馈，用户应预览文件后自行决定是否发送
- **MCP 接入授权**: 任何外部 AI 客户端必须在 App 内显式授权才能读写数据库

如果你发现任何违反以上原则的实现,请按上面流程报告。
