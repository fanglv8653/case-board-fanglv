重新抽取案件里某一份**源文档**(原始上传的 PDF / Word / 图片等),重跑 OCR + 字段抽取,把它的文字内容刷新进抽取库。等同于源文件列表里那个「重新抽取」按钮,只是让你(AI)也能触发。

【何时调用】仅在下列情况、且是**用户希望**时调用:
- 某份文档此前抽取**失败**(extraction_status=failed,如之前因解析报错没出文本),用户要你重试。
- 某份文档当初被**跳过抽取**(没有可读文本,read_case_doc / find_in_document 报「还没抽取过文字」),而现在确实需要它的内容来做分析/写材料。
- 用户明确说「重新抽取 / 重新识别 / 再扫一遍这份文件」。

【重要约束 —— 烧积分,不要擅自调用】
- 重抽会**重跑 OCR 和 LLM**:PDF / 扫描件走云端 OCR 会**消耗用户的 MinerU 积分**(花钱)。
- 因此**不要主动、批量或为了"保险"去重抽**。只在用户提出、或某份文档确实缺文本且用户需要它时,才调用;拿不准就**先问用户是否要重抽**(可以用 ask_user),不要自作主张烧积分。
- 一次只针对**一份**确有必要的文档,不要一次性把整个案件的文档都重抽。

【这是异步后台任务 —— 本轮拿不到结果】
- 本工具只**触发**重抽,立即返回;真正的 OCR/抽取在后台跑,可能要数十秒到几分钟。
- **本轮对话里你无法读到重抽后的新文本**。不要在同一轮里紧接着 read_case_doc 期待拿到新内容 —— 那会读到旧的或仍为空。
- 正确做法:触发后告诉用户「已开始重新抽取,完成后文档文本会更新,源文件列表能看到进度」,让用户等待完成后再让你读取/分析。

【参数】
- doc_id:要重抽的文档标识。可填 list_case_docs 拿到的 id(UUID,最稳),也可直接填文件名(如「离婚补偿协议.pdf」)。AI 生成的文档(分析报告 / 起草的文书)没有原始文件,不能重抽,会被拒绝。

【返回】触发成功返回被重抽文档的名称与 doc_id,以及"后台异步、本轮读不到、稍后再读"的提示。把这层意思如实转达给用户,不要假装已经拿到新内容。
